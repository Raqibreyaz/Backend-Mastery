In AWS Lambda, when a function returns a response, there is a payload size limit of about 6 MB. If the response exceeds this limit, Lambda returns a 413 error indicating that the payload is too large.

By default, Lambda treats the response body as UTF-8 encoded text because the Lambda runtime sits behind an AWS managed server written in low level languages like C or C++, which automatically converts responses to UTF-8. Due to this behavior, returning binary data such as images, PDFs, or videos directly can corrupt the data because Lambda will try to interpret it as text. To safely return binary content, the data must be Base64 encoded and the isBase64Encoded flag must be set to true. When this flag is enabled, Lambda understands that the response body represents binary data and handles it correctly.

Base64 encoding increases the size of the data by roughly 33 percent, meaning every 3 bytes of raw data become 4 bytes after encoding. Because of this size increase and the payload limit, Lambda functions are generally not used for serving files. Although the payload limit can be effectively increased by using response streaming, Lambda is still not ideal for file delivery. AWS provides dedicated services like Amazon S3 and CloudFront for serving static and large files efficiently at scale. Lambda is best suited for handling application logic and returning relatively small JSON responses, where the 6 MB payload limit is more than sufficient for managing APIs and server side processing.
