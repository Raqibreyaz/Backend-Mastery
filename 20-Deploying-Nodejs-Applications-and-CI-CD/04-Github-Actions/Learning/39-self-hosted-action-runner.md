1. A self-hosted runner is a machine you manage that executes GitHub Actions jobs instead of GitHub-hosted infrastructure. [docs.github](https://docs.github.com/actions/hosting-your-own-runners)
2. You use it when you need more control over OS, hardware, installed tools, network access, or custom environments. [github](https://github.blog/enterprise-software/ci-cd/when-to-choose-github-hosted-runners-or-self-hosted-runners-with-github-actions/)
3. It is useful for large builds, private dependencies, on-prem resources, or workloads that should stay inside your own infrastructure. [docs.github](https://docs.github.com/en/actions/concepts/runners/self-hosted-runners)
4. Setup starts by creating a runner in repo/org settings, downloading the runner package, and configuring it with the provided registration token. [dev](https://dev.to/s3cloudhub/step-by-step-guide-to-configuring-a-self-hosted-runner-in-github-actions-2024-2b7j)
5. After configuration, you start the runner manually or as a service so it stays available for jobs. [docs.github](https://docs.github.com/actions/hosting-your-own-runners/managing-self-hosted-runners/configuring-the-self-hosted-runner-application-as-a-service)
6. In workflow YAML, you target it with `runs-on: self-hosted` or with labels like `runs-on: [self-hosted, linux, x64]`. [docs.github](https://docs.github.com/en/actions/how-tos/manage-runners/self-hosted-runners/use-in-a-workflow)
7. GitHub sends the job to the runner, and the runner executes each step on your machine while staying connected to GitHub. [github](https://github.com/actions/runner)
8. The main tradeoff is control versus responsibility: you gain flexibility, but you must maintain the machine, updates, and availability yourself. [docs.github](https://docs.github.com/actions/hosting-your-own-runners)
